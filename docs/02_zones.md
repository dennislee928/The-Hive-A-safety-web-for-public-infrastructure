# Zones

本文件定義四個高密度區域的詳細建模規格。每個區域具有獨特的拓撲特性、信號來源配置、決策點適用性與隱私考量。

## Z1 — Station Interior

### 拓撲模型

**空間結構**：

- **大廳層（Concourse）**：票閘區、服務台、商業空間、轉乘通道入口
- **月台層（Platform）**：候車區、上下車區域、緊急出口、電梯/手扶梯
- **轉乘層（Transfer）**：跨線轉乘通道、垂直動線（電梯、手扶梯、樓梯）

**關鍵瓶頸點**：

- 票閘機群（進出站流量集中）
- 手扶梯/電梯（垂直動線限制）
- 月台候車區（列車到站前後的人流聚集）
- 轉乘通道（多線交會處的交叉流量）

**流量模式**：

- **正常時段**：雙向穩定流，高峰時段有明顯方向性
- **緊急疏散**：單向流出為主，需避免對向衝突
- **部分封閉**：繞行路徑的重新分配與容量限制

**疏散路徑優先順序**：

1. 主要出口（票閘區 → 地面層）
2. 緊急出口（月台層直接通道）
3. 轉乘通道（利用鄰近站點疏散）
4. 服務通道（工作人員引導）

### 感測器配置與信號來源

**固定基礎設施信號**：

- 票閘流量計數（進出站人次，時間窗口聚合）
- 手扶梯/電梯運行狀態與負載感測
- 月台監控系統（人群密度影像分析，僅區域級聚合）
- 環境感測器（煙霧、溫度、異常震動）

**人員信號**：

- 站務人員回報（結構化報告：位置、觀察、時間戳）
- 保全人員巡邏狀態與異常通報

**群眾信號（Route 2）**：

- 結構化報告（區域：Z1，子區域：大廳/月台/轉乘，時間窗口，信心度）
- 僅在時間窗口內聚合，不追蹤個別裝置軌跡

**信號聚合規則**：

- 時間窗口：60 秒（可調整）
- 空間粒度：子區域級（大廳/月台/轉乘），不進行精確定位
- 信號加權：基礎設施（0.4）、人員（0.4）、群眾（0.2，需佐證）

### 決策點適用性

- **D0（Pre-Alert）**：適用，可監控多個子區域
- **D1（Dispatch）**：適用，建議資源配置（站務/保全/第一線應變）
- **D2（Local Guidance）**：適用，PA/電子看板/App 指引
- **D3（Zone Escalation）**：適用，可針對單一子區域或全站內
- **D4（Multi-Zone）**：適用，與 Z3 協調（站內外聯動）
- **D5（Public Warning）**：適用，站內廣播 + 外部通道
- **D6（De-escalation）**：適用，狀態回滾與證據封存

### 區域間協調需求

- **與 Z2（列車）**：列車到站前的預警協調，月台層疏散指引
- **與 Z3（站周邊）**：出口流量控制，避免站外擁堵回溢
- **與 Z4（其他高密度區）**：大型活動時的站內容量管理

### 隱私與安全考量

**資料最小化**：

- 不儲存個別人員的進出站記錄
- 僅保留區域級聚合統計（時間窗口摘要）
- 影像分析僅用於密度估算，不進行人臉識別或個別追蹤

**存取控制**：

- 即時監控資料：僅授權操作人員（需雙因子認證）
- 歷史聚合資料：去識別化後用於分析，保留期限 90 天
- 審計日誌：所有資料存取記錄，保留 1 年

**攻擊考量**：

- 感測器偽造：需硬體完整性驗證與跨信號佐證
- 群眾報告濫用：速率限制（每裝置每小時最多 3 筆），需與基礎設施信號佐證

## Z2 — Train Car

### 拓撲模型

**空間結構**：

- **車廂內部**：座位區、站立區、車門區、緊急通話裝置
- **列車編組**：多節車廂連接，車廂間通道（如有）

**關鍵瓶頸點**：

- 車門區域（上下車集中點）
- 車廂間通道（窄通道，流量受限）
- 緊急出口（僅特定車廂配置）

**流量模式**：

- **正常營運**：到站前後的人流移動（上車/下車）
- **緊急停車**：可能發生在站間，疏散需等待到站或緊急通道開啟
- **部分車廂封閉**：其他車廂的容量重新分配

**疏散選項**：

1. **到站疏散**（優先）：列車停靠最近站點，利用 Z1 資源
2. **緊急通道**：特定車廂的緊急出口（需工作人員操作）
3. **車廂間移動**：引導至安全車廂，等待到站

### 感測器配置與信號來源

**固定基礎設施信號**：

- 列車運行狀態（位置、速度、緊急停車狀態）
- 車門開關狀態與異常
- 車廂環境感測（煙霧、溫度、異常震動）
- 緊急通話裝置觸發（位置：車廂編號）

**人員信號**：

- 列車駕駛員回報（結構化：車廂、觀察、時間戳）
- 隨車服務人員（如有）的巡視回報

**群眾信號（Route 2）**：

- 結構化報告（區域：Z2，子區域：車廂編號，時間窗口，信心度）
- 緊急求助請求（最小資料：車廂編號、需求類型）

**信號聚合規則**：

- 時間窗口：30 秒（列車環境變化較快）
- 空間粒度：車廂級（不進行車廂內精確定位）
- 信號加權：基礎設施（0.5）、人員（0.4）、群眾（0.1，需強佐證）

### 決策點適用性

- **D0（Pre-Alert）**：適用，監控特定車廂或整列車
- **D1（Dispatch）**：適用，建議到站應變資源配置
- **D2（Local Guidance）**：適用，車廂內廣播/顯示器/App 指引
- **D3（Zone Escalation）**：適用，可針對單一車廂或整列車（需與 Z1 協調）
- **D4（Multi-Zone）**：適用，列車與到站 Z1 的協調
- **D5（Public Warning）**：適用，列車內廣播 + 到站後 Z1 通道
- **D6（De-escalation）**：適用，狀態回滾與證據封存

### 區域間協調需求

- **與 Z1（站內）**：到站前的預警通知，月台層疏散準備，資源調度
- **與其他 Z2（鄰近列車）**：網路級協調，避免多列車同時到站造成 Z1 擁堵

### 隱私與安全考量

**資料最小化**：

- 不追蹤個別乘客的上下車記錄
- 僅保留車廂級聚合統計（時間窗口摘要）
- 緊急通話資料：僅保留事件相關記錄，保留期限 30 天

**存取控制**：

- 即時監控資料：列車控制中心授權人員
- 歷史聚合資料：去識別化後用於分析，保留期限 60 天
- 審計日誌：所有資料存取記錄，保留 1 年

**攻擊考量**：

- 列車系統入侵：需硬體完整性驗證與網路隔離
- 群眾報告濫用：速率限制（每裝置每小時最多 2 筆），需與列車狀態信號佐證

## Z3 — Station Perimeter

### 拓撲模型

**空間結構**：

- **站體周邊**：主要出入口、次要出入口、無障礙通道
- **地面層介面**：人行道、公車轉乘站、計程車排班區、自行車停車區
- **街道路網**：鄰近道路交叉口、斑馬線、天橋/地下道

**關鍵瓶頸點**：

- 主要出入口（站內外流量交會）
- 公車轉乘站（多路線匯集，候車區容量限制）
- 道路交叉口（行人與車輛交會，需交通號誌協調）
- 天橋/地下道（垂直動線，容量受限）

**流量模式**：

- **正常時段**：進出站雙向流，轉乘流（公車/計程車）
- **緊急疏散**：站內向外的單向流出，需避免與進站人流衝突
- **大型活動**：額外聚集與疏散需求，需多單位協調

**疏散路徑優先順序**：

1. 主要出口 → 鄰近安全集結點
2. 次要出口 → 替代路徑
3. 公車轉乘站 → 緊急疏散公車（如有）
4. 鄰近開放空間（公園、廣場）

### 感測器配置與信號來源

**固定基礎設施信號**：

- 出入口流量計數（進出人次，時間窗口聚合）
- 公車轉乘站監控（候車人數，區域級聚合）
- 道路監控系統（車流、人流密度，區域級聚合）
- 環境感測器（異常事件偵測）

**人員信號**：

- 站外保全/巡邏人員回報
- 公車站務人員回報（如有）
- 交通管理單位協調（大型活動時）

**群眾信號（Route 2）**：

- 結構化報告（區域：Z3，子區域：出入口/轉乘站/道路，時間窗口，信心度）
- 緊急求助請求（最小資料：大致位置、需求類型）

**信號聚合規則**：

- 時間窗口：90 秒（周邊環境變化較慢，但需考慮交通流）
- 空間粒度：子區域級（出入口/轉乘站/道路段），不進行精確定位
- 信號加權：基礎設施（0.35）、人員（0.35）、群眾（0.3，需佐證）

### 決策點適用性

- **D0（Pre-Alert）**：適用，監控多個子區域
- **D1（Dispatch）**：適用，建議資源配置（保全/交通管理/第一線應變）
- **D2（Local Guidance）**：適用，電子看板/App 指引、交通號誌協調
- **D3（Zone Escalation）**：適用，可針對單一子區域或全周邊
- **D4（Multi-Zone）**：適用，與 Z1/Z4 協調（站內外聯動、大型活動）
- **D5（Public Warning）**：適用，區域廣播 + 交通管理協調
- **D6（De-escalation）**：適用，狀態回滾與證據封存

### 區域間協調需求

- **與 Z1（站內）**：出口流量控制，避免站外擁堵回溢至站內
- **與 Z4（其他高密度區）**：大型活動時的周邊交通管理與疏散協調
- **與外部單位**：交通管理、公車營運、緊急應變單位

### 隱私與安全考量

**資料最小化**：

- 不追蹤個別人員的進出站記錄或移動軌跡
- 僅保留區域級聚合統計（時間窗口摘要）
- 影像分析僅用於密度估算，不進行人臉識別或個別追蹤

**存取控制**：

- 即時監控資料：授權操作人員與協調單位（需雙因子認證）
- 歷史聚合資料：去識別化後用於分析，保留期限 90 天
- 審計日誌：所有資料存取記錄，保留 1 年

**攻擊考量**：

- 感測器偽造：需硬體完整性驗證與跨信號佐證
- 群眾報告濫用：速率限制（每裝置每小時最多 3 筆），需與基礎設施信號佐證
- 交通系統干擾：需與交通管理單位建立安全通訊通道

## Z4 — Other High-Density Areas

### 拓撲模型

**空間結構**：

- **活動場地**：室內/室外活動空間、舞台區、觀眾區、出入口
- **廣場/公園**：開放空間、聚集點、疏散路徑
- **商業區**：購物中心、夜市、市集、街道

**關鍵瓶頸點**：

- 主要出入口（活動場地/商業區）
- 舞台/核心區域（人群聚集中心）
- 疏散通道（窄通道、樓梯、出口）
- 停車場/轉乘點（車輛與人流交會）

**流量模式**：

- **活動進行中**：向心聚集（朝向舞台/核心區）
- **活動結束/緊急疏散**：離心擴散（朝向出口/安全區）
- **部分區域封閉**：繞行路徑的重新分配

**疏散路徑優先順序**：

1. 主要出口 → 鄰近開放空間
2. 次要出口 → 替代路徑
3. 緊急通道 → 工作人員引導
4. 鄰近安全集結點（公園、廣場、避難所）

### 感測器配置與信號來源

**固定基礎設施信號**：

- 活動場地監控系統（人群密度，區域級聚合）
- 出入口流量計數（進出人次，時間窗口聚合）
- 環境感測器（煙霧、溫度、異常震動）
- 活動管理系統（活動狀態、參與人數估算）

**人員信號**：

- 活動工作人員回報（結構化：位置、觀察、時間戳）
- 保全/巡邏人員回報
- 協調單位（警察、消防、醫療）回報

**群眾信號（Route 2）**：

- 結構化報告（區域：Z4，子區域：活動場地/廣場/商業區，時間窗口，信心度）
- 緊急求助請求（最小資料：大致位置、需求類型）

**信號聚合規則**：

- 時間窗口：120 秒（活動環境變化較慢，但需考慮突發事件）
- 空間粒度：子區域級（活動場地/廣場/商業區），不進行精確定位
- 信號加權：基礎設施（0.3）、人員（0.4）、群眾（0.3，需佐證）

### 決策點適用性

- **D0（Pre-Alert）**：適用，監控多個子區域
- **D1（Dispatch）**：適用，建議資源配置（保全/第一線應變/醫療）
- **D2（Local Guidance）**：適用，廣播/電子看板/App 指引
- **D3（Zone Escalation）**：適用，可針對單一子區域或全區域
- **D4（Multi-Zone）**：適用，與 Z1/Z3 協調（大型活動時的站區聯動）
- **D5（Public Warning）**：適用，區域廣播 + 多通道發布
- **D6（De-escalation）**：適用，狀態回滾與證據封存

### 區域間協調需求

- **與 Z1/Z3（站區）**：大型活動時的站區容量管理與疏散協調
- **與外部單位**：活動主辦單位、警察、消防、醫療、交通管理

### 隱私與安全考量

**資料最小化**：

- 不追蹤個別參與者的移動軌跡
- 僅保留區域級聚合統計（時間窗口摘要）
- 活動參與資料：僅在活動期間保留，活動結束後 7 天內刪除

**存取控制**：

- 即時監控資料：授權操作人員與協調單位（需雙因子認證）
- 歷史聚合資料：去識別化後用於分析，保留期限 60 天
- 審計日誌：所有資料存取記錄，保留 1 年

**攻擊考量**：

- 感測器偽造：需硬體完整性驗證與跨信號佐證
- 群眾報告濫用：速率限制（每裝置每小時最多 2 筆），需與基礎設施信號佐證
- 活動系統入侵：需網路隔離與安全通訊通道

## 區域間協調框架

### 協調原則

1. **資訊共享**：區域間共享聚合後的狀態摘要（不共享原始個別資料）
2. **決策優先順序**：高影響決策（D3/D4/D5）需跨區域確認
3. **資源調度**：可動態分配資源（人員、設備）至不同區域
4. **一致性保證**：所有區域的公開指引需保持一致（基於同一 CAP 訊息）

### 協調機制

- **狀態同步**：每 30 秒交換區域狀態摘要（聚合後資料）
- **決策通知**：高影響決策（D3/D4/D5）需通知相關區域
- **資源請求**：區域可請求其他區域的資源支援（需授權）

### 假設

- 區域間通訊通道安全且可靠
- 區域時鐘同步（時間戳一致性）
- 區域操作人員具備跨區域協調權限（需授權）

### 攻擊考量

- 區域間通訊偽造：需加密與完整性驗證
- 跨區域資源濫用：需授權機制與審計日誌

### 隱私影響

- 區域間僅共享聚合資料，不共享原始個別資料
- 跨區域分析需去識別化處理
